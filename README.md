## Event System Origins

In modern web application systems, it's common for data produced by a business process to need to **"flow through"** the entire web system/API. This is often used for data collection, monitoring, distribution, and more — as well as for building associations with other systems (e.g., constructing a call chain to control other system features).

## How the Event System Works

Similar to an MVC framework, there is an entry point (here referred to as the event trigger point). From there, events are handed off to different dispatchers, which route them to various handler layers. Finally, the event is received, processed, and consumed.

## About Features 

- Implements automatic loading of a given directory, consistent with PSR-4 conventions

	- event                     `` ( Core of the event system )``  
		+ Utility               `` ( Helper classes for the event system) ``
		+ ..
	- tracker					`` ( Event system trace line ) `` 
	- config                    `` ( Behavioral configuration for the event system )`` 
	- logs						`` ( Log data generated by the event system )`` 

- Implements event data stream reception
    + An event is composed of two parts: ``EventData,  EventCore ``
    + Event data can currently be generated by passing an array
    + Event data implements the magic method ``__toString``, so it can be directly output
    + Events are received and then **tracked/processed** by the ``EventDispatcher ``

- Implements event dispatching and data monitoring
    + There are currently three types of dispatch rules: **generic events (wildcard name), specific name, and specific action**
    + Each dispatch rule is bound to a tracker (under the EventTracker directory) 
    + By default, all events fall under the generic event category, represented by **``*``** in the configuration file. Therefore, the tracker bound to the generic event can receive event data from any source (useful for monitoring, call chain tracing, etc.)）
    + A tracker bound to a specific named event will only receive data for that particular event; data from other events will not be captured
    
## Usage：

- This event system can be directly integrated with native PHP applications
- The file ``autoloader.php `` in the root directory implements the system's loading rules — simply include it
- If you are using Composer, it may conflict with the existing ``spl_autoload_register`` mechanism. You will need to modify the loading rules accordingly; refer to the source code for details
- The system uses a .env configuration file (just copy .env.example to .env)
- By default, the system enables trace logging (this can be disabled via the configuration file)
- Custom configuration entries can be defined in ``config/settings.php``
- Usage examples are provided in index.php
